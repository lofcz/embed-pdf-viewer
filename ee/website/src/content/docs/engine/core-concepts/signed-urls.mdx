# Signed URLs

CloudPDF never serves raw file links to the browser. Instead, your server mints
a **signed URL** — a short-lived, cryptographically signed link scoped to a
single document and a single viewer session.

## Why signed URLs

- **Expiry** — links stop working after a window you choose.
- **Per-document scope** — a URL grants access to exactly one document.
- **Permission flags** — toggle `download` and `print` per link.
- **Revocability** — rotate a key or delete the document to invalidate links.

## Minting a URL

```ts
const { url, expiresAt } = await cloudpdf.documents.sign({
  documentId: 'doc_3kZ…',
  expiresIn: '15m',
  permissions: { download: false, print: false },
  viewer: { id: 'user_2231', email: 'dana@acme.com' },
});
```

The optional `viewer` object is attached to every audit event generated by this
session, so you can attribute opens and downloads to a specific person.

## Choosing an expiry

Short windows are safer but require your client to refresh. A common pattern:

- **15m** for one-off document views.
- **1h** for an editing session.
- **5m** for highly sensitive documents behind an extra confirmation step.

<Callout type="info">
  The viewer automatically requests a fresh URL before the current one expires,
  as long as your sign endpoint is reachable. You rarely need to manage refresh
  yourself.
</Callout>

## Verifying requests server-side

Every signed URL is validated at the edge on each request. A tampered or
expired URL is rejected before any page data is streamed.

<Callout type="warn">
  Don't cache signed URLs in a shared store or CDN. They are scoped to a single
  viewer and a short window — caching defeats both protections.
</Callout>
